欢迎!本指南包含编写和维护 Prisma 工具技术文档的信息。本风格指南的一个主要目标是确保 Prisma 文档在选词、写作风格、格式等方面的一致性。

> **本风格指南很大程度上来自于 [Gatsby 风格指南](https://www.gatsbyjs.org/contributing/gatsby-style-guide/).** 许多章节都被直接复制了，有些章节经过了轻微的编辑，还有一些是新的，是 Prisma 文档所特有的。

</TopBlock>

## 选词

### 用“你”作代词

任何内容都应该使用第二人称(“你”)来提供对话基调。这样，文本和说明似乎直接与阅读它的人对话。尽量避免使用第一人称(“我”、“我们”、“让我们”和“我们”)。

使用英语中的“你”也比说“我”更准确，因为通常一次只有一个人在阅读教程或指南，而编写教程的人实际上并没有和他们一起阅读，所以“我”是不准确的。你可能会注意到，一些技术文档使用了第三人称代词和名词，比如“他们”和“用户”，它们比对话中温暖的“你”和“你的”增加了更多的距离，让人感觉更冷。

#### 什么时候使用"我们"

当你特别提到 Prisma 时，你可以用“我们”。例如：

> "我们建议你在你的应用程序中共享一个`PrismaClient`实例。

### 避免使用像“容易”和“公正”这样的字眼。

避免使用“容易”、“只是”、“简单”和“基本”这样的词，因为如果用户很难完成本应“容易”的任务，他们就会质疑自己的能力。考虑使用更具体的描述符；例如，当你说“部署很容易”这句话时，你的真正意思是什么？这是不是因为它比另一种选择需要更少的步骤而容易呢？如果是，请尽可能使用最具体的描述符，在这种情况下应该是“此部署方法比其他选项涉及的步骤更少”。

对于更具包容性的文档，避免使用假定读者的经验或技能水平的短语，如“只需部署它，你就完成了”或“为一个复习人员(指的是一个完全不同的文档，某人可能还没有读过)”。通常情况下，重新措辞会产生更强烈的句子，从而吸引更广泛的上下文。

### 使用包容性语言

当你需要用第三人称来指代一个或多个人时，如果这个人的性别不明确，一定要使用包容的、中性的语言。用“他们/他们/他们的”代替“他/他/他的”或“她/她/她”。避免使用像“伙计们”这样的字眼。

### 避免使用表情符号、粗话和隐喻

避免在文档和习惯用语/粗语或隐喻中使用表情符号或情绪。Prisma 有一个全球社区，表情符号、表情符号或粗语的文化含义可能在世界各地有所不同。用你最好的判断力！此外，表情符号在不同的系统上可以呈现不同的效果。

### 避免使用拉丁语术语

举例:

- et al
- etc
- i.e.
- e.g.

### 避免缩写

举例:

- You're (use "You are")
- It'll (use "It will")

### 定义行话

文章应该用简短、清晰的句子写，必要时尽量少用行话。

> 行话：(名词)某一特定职业或群体使用的、其他人难以理解的特殊词语或表达：法律术语。

所有行话都应该立即用通俗易懂的英语来定义。换句话说，假装你的读者有基本的编码经验，但不一定有使用 PWAS 和 JAMstack 的经验(请看那里发生了什么？我只用了两个需要定义的行话)；你需要定义新手可能很难理解的单词。

### 参考文档的其他部分

- Page
- Section

### 技术用词选择

#### 记录

将数据库中的行称为**记录**。例如：:

"下面的 `create` 查询创建一个 `User` **记录**."

不用:

- Entry
- Row
- Object

#### 模型特性

模型属性是指引用模型的顶级`PrismaClient`属性：:

```
const result = await prisma.user.findMany(...) // user model property
const result = await prisma.post.findMany(...) // post model property
```

## 写作风格

### 写得简明扼要，避免长篇大论

简洁明了的写作传达了最低限度的信息，没有冗余。努力使你的写作尽可能简短；这样的练习通常会让你的写作更准确、更具体。要避免大块文本，请执行以下操作：

- 使用较短的段落
- 使用列表
- 使用示例
- 使用粗体和强调
- 使用表格
- 使用图表

### 使用主动语态

用主动语态代替被动语态。一般来说，这是一种更简洁、更直截了当的交流方式。例如：

- (被动)程序员使用 javascript 中的 for 循环来…
- (主动)程序员在 javascript 中使用 for 循环…

### 要有主见

使用肯定的语言。

推荐:

- 使用`createMany`方法在单个事务中创建多条记录
- 你可以使用嵌套写入同时创建 user 和该用户的 posts

避免:

- 例子中尝试..
- 你可能..
- 你应该使用..

### 使用清晰的超链接

超链接应该包含最清晰的文字，以表明链接将把你带到哪里。

```md
<!-- Good -->

获得更多 [Prisma 文档](https://www.prisma.io/docs/)

<!-- Bad -->

在 Prisma 文档获得更多 [这里](https://www.prisma.io/docs/)
```

### 指示何时某些内容是可选的

当一个段落或句子提供可选路径时，第一句话的开头应该表明它是可选的。例如，“如果你想了解更多关于 xyz 的信息，请参阅我们的参考指南”比“如果你想了解更多关于 xyz 的信息，请转到参考指南”更清楚。

这种方法可以让不想更多了解 xyz 的人尽早停止阅读句子。这种方法还可以让想要了解更多 xyz 的人认识到更快学习的机会，而不是不小心跳过段落。

### 缩写术语

如果你想在你的文章中缩写一个术语，先把它完整地写出来，然后把缩写放在括号里。在此之后，您可以在文章的其余部分使用缩写。例如，“在计算机科学中，抽象语法树(AST)是…”。

### 拥抱冗余

在用户所在的地方与他们见面。

## 语法和格式

### 专有名词大写拼写

尽管在编写“Javascript”(而不是“JavaScript”)时使用诸如“Postgres”(而不是官方专有名词“PostgreSQL”)之类的缩写很有诱惑力，或者不必担心大小写问题，但我们致力于使用专有名词的首选概念。以下是几个常见的示例：

- Node.js (instead of Node or Node.JS)
- JavaScript and TypeScript (instead of JavaScript and Typescript or JS and TS)
- PostgreSQL (instead of Postgres or Postgresql)
- MongoDB (instead of Mongo)
- GitHub (instead of Github)
- ...

如果你不确定某个名词的拼写，它的官方主页通常包含正确的拼写供你查阅。

### 标题和页眉

- 标题和页眉使用句子大小写：只有首个单词是大写的
- 使用动词的现在分词形式(添加**-ing**-例如：“配置数据库”)
- 除非需要问号，否则不要在短语末尾使用标点符号
- 标题中的代码使用`<inlinecode>code</inlinecode>`-这是导航元素所必需的

### 使用表格、项目符号列表和编号列表

表格和列表通常是表示信息的最简明的方式。只要它们觉得合适，就可以使用这些元素。以下是一些指导原则，可帮助你决定何时使用以下哪种方法：

- 仅在列表的顺序无关紧要时才使用项目符号列表(例如，没有固有顺序的数据库功能的枚举)。
- 仅当列表的顺序很重要时才使用编号列表(例如，在提供一个步骤建立在前一个步骤基础上的分步说明时)。
- 当你枚举具有许多相似属性/特征的事物时(例如，API 调用的参数都有一个“名称”、一个“类型”，是必需的或可选的，并且需要一个描述)，请使用表格。

### 使用斜体和粗体

斜体和粗体可以很好地向读者强调句子的某些部分。只有当你想让这一部分从整个段落中突出时，才能使用更稀疏的粗体(这样，当读者只“浏览”页面而不是正确阅读时，它才是可见的)。首次引入新概念的单词使用斜体。

### 对路径和文件名使用内联代码格式

举例:

- “生成的 Prisma 客户端默认位于`./node_module/.prisma`文件夹中。”
- “`schema.prisma`文件位于...”
- “要使用多个`.env`文件...”

### 引用文本中的字符串时使用内联代码格式

举例:

“下面的查询返回`email`字段等于`Sarah`的所有记录。”

### 用牛津逗号清楚地列出清单

除标题外，请使用牛津逗号。它是一个逗号，用在三个或更多项目列表中倒数第二个项目之后，在“and”或“or”之前，例如意大利画家、雕塑家和建筑师。这让事情变得更清晰了。

[如果不使用牛津逗号，可能会发生混淆。](https://img.buzzfeed.com/buzzfeed-static/static/2015-02/22/18/enhanced/webdr11/enhanced-buzz-32156-1424646300-12.jpg?downsize=715:*&output-format=auto&output-quality=auto)

### 首选美式英语

对于有多个拼写的单词，首选美国英语单词，而不是英国英语或加拿大英语。例如：

- "color" 代替 "colour"
- "behavior" 代替 "behaviour"

## 编写代码片段

### 介绍所有代码片段

写一句简短的开场白：

- 解释代码段正在执行的操作。
- 指向参考文档的链接(如果适用)

举例:

```md
The following [`createMany`](..) query creates several new `User` records:
```

```md
The following [`createMany`](..) query:

- Creates several `User` records
- Creates several nested `Post` records
- Creates several nested `Category` records
```

````
```prisma highlight=3;normal
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["orderByRelations"]
}
```
````

### 强调占位符

占位符在技术文档中是一个棘手的主题，也是最常见的混淆来源之一，尤其是对初学者而言。为了保持一致性，Prisma 文档中的占位符为：

- 全部用大写字母拼写
- 前缀和后缀分别为两个下划线
- 使用描述性术语

例如，考虑以下代码块，其中`__DATABASE_CONNECTION_URL__`是 PostgreSQL 连接 URL 的占位符：

```prisma
datasource db {
  provider = "postgresql"
  url      = "__DATABASE_CONNECTION_STRING__"
}
```

无论何时使用占位符，都应该确保直接解释或至少链接到另一个解释如何获取占位符的值的资源。您还应该明确指出，这是一个占位符，需要替换为“实际值”，包括该实际值可能是什么样子的示例：

```prisma
datasource db {
  provider = "postgresql"
  url      = "postgresql://opnmyfngbknppm:XXX@ec2-46-137-91-216.eu-west-1.compute.amazonaws.com:5432/d50rgmkqi2ipus?schema=hello-prisma2"
}
```

### Shell 命令列表

当你需要向读者提供一系列 CLI 命令作为说明时，除非您为每个步骤提供上下文，否则不要使用列表：

#### 避免

- `cd path/to/server`
- `docker-compose up -d --build`
- `./node_modules/.bin/sequelize db:migrate` or `npx sequelize db:migrate`
- `./node_modules/.bin/sequelize db:seed:all` or `npx sequelize db:seed:all`

#### 推荐

```shell
cd path/to/server
docker-compose up -d --build
./node_modules/.bin/sequelize db:migrate # or `npx sequelize db:migrate`
./node_modules/.bin/sequelize db:seed:all # or `npx sequelize db:seed:all`
```

或者

1. 切换目录: `cd path/to/server`
1. 启动容器: `docker-compose up -d --build`
1. 迁移数据库结构: `./node_modules/.bin/sequelize db:migrate` or `npx sequelize db:migrate`
1. 数据库种子数据: `./node_modules/.bin/sequelize db:seed:all` or `npx sequelize db:seed:all`

````
```terminal
npm install prisma

````

For example:


## 拼写

### 连字符

根据这些 [规则](https://www.grammarbook.com/punctuation/hyphens.asp)使用连字符。

有时有些术语不清楚是否使用连字符。为了保持一致性，我们在这里列出了这些术语，作为在整个 Prisma 文档中使用它们的规则。

#### 不带连字符的拼写

- 使用案例
- 命令行
- 自动套用格式
- 类型安全

#### 带连字符的拼写

- 编译时间

#### 拼写为一个单词

- 自动完成

#### 类型安全和安全类型

- "这个代码是安全类型"
- "这个是类型安全代码"

#### 数据源和 `datasource`

- `datasource` 区块
- “引入新数据源时必须重新生成客户端”
